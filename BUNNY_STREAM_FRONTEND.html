EduTrack Online - Bunny Stream API

Target Audience: Next.js Frontend Developers Library ID: 698152 CDN Hostname: vz-964b4290-bbe.b-cdn.net

Overview

All videos on EduTrack are hosted on Bunny Stream. The backend never stores video files — it relays uploads to Bunny and generates signed playback URLs.

Upload Flow — Full File

Teacher Browser → `POST /courses/videos/<id>/upload/` (file) → Backend streams to Bunny CDN

Best for small files (<100 MB) on stable connections. Single PUT, no pause/resume.

Upload Flow — Chunked (Pause/Resume)

Teacher Browser → chunk 0  → `POST /courses/videos/<id>/upload/` (file + offset + total_size)
                → chunk 1  → `POST /courses/videos/<id>/upload/` ...
                → chunk N  → Backend PUTs each chunk with Content-Range header → Bunny CDN
                ↕ pause/resume
                → `GET /courses/videos/<id>/upload-status/` → Backend queries Bunny → returns uploaded_bytes

Frontend splits the file into fixed-size chunks (recommended: 5 MB). Each chunk is sent via POST /upload/ with an offset field. If the connection drops, the frontend checks how many bytes Bunny has received and resumes from that offset. When the user clicks Pause, the frontend simply stops sending chunks and remembers offset; later it can resume by querying upload-status and continuing.

The file never sits on the backend disk — each chunk is read and streamed directly to Bunny.

Playback Flow

Student Browser → `GET /courses/videos/<id>/play/` → Backend checks permissions → Returns signed HLS URL
                                            ↘ Student streams directly from Bunny CDN (0 server load)

2. Upload Flow

POST /courses/videos/<id>/create-upload/

Description: Creates a video entry on Bunny Stream and returns the Bunny GUID. Must be called before uploading. The returned upload_url is for backend use only — the frontend should upload via the proxy endpoint instead.

Authentication: Teacher (must own the lecture's course)

Request Body: None

Success Response — 201 Created:

{
  "video_id": "integer - Local video ID",
  "bunny_video_id": "string - Bunny Stream GUID",
  "upload_url": "string - Bunny CDN upload URL (backend use only)",
  "status": "integer - Bunny status (0 = Created)"
}

POST /courses/videos/<id>/upload/

Description: Upload a video file (or chunk) to Bunny Stream via the backend proxy.

Two modes:

Mode offset field Supports pause/resume Use case
Full file absent ❌ ❌ Files <100 MB on stable connections
Chunked present ✅ ✅ Large files, slow networks, pause/resume needed

Authentication: Teacher (must own the lecture's course)

Content-Type: multipart/form-data

Mode A — Full file upload (backward-compatible)

Single request, entire file.

Field Type Required Description
file File ✅ Video file (any format supported by Bunny Stream)

Success Response — 200 OK:

{
  "success": true,
  "video_id": "integer",
  "bunny_video_id": "string",
  "bunny_status": 1,
  "bunny_status_display": "Uploaded",
  "message": "File uploaded to Bunny Stream successfully. Encoding will begin shortly."
}

Mode B — Chunked upload (pause/resume)

Frontend splits the file into fixed-size chunks (recommended: 5 MB / 5242880 bytes). Each chunk is sent in a separate request. The backend PUTs it to Bunny with a Content-Range header — Bunny stores the chunk at the correct byte position.

Field Type Required Description
file File ✅ The chunk binary data
offset Integer ✅ Byte offset of this chunk (0-based)
total_size Integer ✅ Total file size in bytes

Success Response — 200 OK:

{
  "success": true,
  "video_id": "integer",
  "bunny_video_id": "string",
  "offset": "integer — byte offset of the chunk just uploaded",
  "chunk_size": "integer — bytes in this chunk",
  "total_size": "integer — total file size",
  "is_last": "boolean — true if this was the final chunk",
  "bunny_status": "integer (0 = still uploading, 1 = complete)",
  "bunny_status_display": "Created | Uploaded"
}

How pause/resume works:

  1. Pause — Frontend stops sending chunks. Bunny keeps the already-received bytes. No action needed on the backend.

  2. Resume — Frontend calls GET /courses/videos/<id>/upload-status/ to get uploaded_bytes (how many bytes Bunny has stored so far). Then it sends chunks starting at offset = uploaded_bytes.

  3. Interruption — If the internet drops mid-upload, same as pause: check upload-status and resume from uploaded_bytes.

Chunk upload example (frontend JavaScript):

const CHUNK_SIZE = 5 * 1024 * 1024; // 5 MB
const file = selectedFile; // File object
const totalSize = file.size;

for (let offset = 0; offset < totalSize; offset += CHUNK_SIZE) {
  // Check if user paused
  if (paused) break;

  const chunk = file.slice(offset, offset + CHUNK_SIZE);
  const formData = new FormData();
  formData.append('file', chunk);
  formData.append('offset', offset.toString());
  formData.append('total_size', totalSize.toString());

  const resp = await fetch(`/courses/videos/${videoId}/upload/`, {
    method: 'POST',
    headers: { /* auth */ },
    body: formData,
  });

  if (!resp.ok) {
    // Retry the chunk or check upload-status and resume
    break;
  }
}

// Resume after interruption:
const statusResp = await fetch(`/courses/videos/${videoId}/upload-status/`);
const { uploaded_bytes } = await statusResp.json();
// Continue loop from uploaded_bytes

Error Responses:

Status Error Cause
400 "No Bunny video entry found" create-upload/ was not called first
400 "No file provided" Missing file field in form data
400 "offset and total_size must be integers" Chunked mode with bad offset/total_size values
403 "You can only upload videos to your own courses" Wrong teacher
410 "Bunny video entry no longer exists" Bunny auto-deleted the entry; call create-upload again
502 "Chunk upload to Bunny Stream failed" Bunny CDN rejected the chunk (retry it)
503 "Server is currently busy with other uploads" Temp disk full

GET /courses/videos/<id>/upload-status/

Description: Query Bunny Stream for how many bytes have been successfully uploaded. Used to resume interrupted chunked uploads.

Authentication: Teacher (must own the lecture's course)

Success Response — 200 OK:

{
  "video_id": "integer",
  "bunny_video_id": "string",
  "uploaded_bytes": "integer — bytes Bunny has received so far"
}

Usage: Frontend calls this before resuming a paused/failed upload, then starts sending chunks from uploaded_bytes.

3. Playback

GET /courses/videos/<id>/play/

Description: Get a signed HLS playback URL for a video. Multiple security checks are performed before the URL is returned.

Authentication: Student, Teacher, Assistant, SiteOwner

Access Checks (Student):

Check Blocked?
Video not found or not active 404
Student not authenticated 401
Lecture not purchased 403
Purchase expired 403
Watching limit reached (default: 4) 403
Prerequisite homework not passed 403
Prerequisite quiz not passed 403
Prerequisite exam not passed 403

Teacher/Assistant: Can preview any video in their own courses (skips purchase/prerequisite checks).

Success Response — 200 OK:

{
  "playback_url": "string - Signed HLS URL (expires in 2 hours)",
  "expires_at": "integer - Unix timestamp when the URL expires"
}

Error Responses:

Status Error Cause
404 "Video is not available." Video not found or inactive
401 "Authentication credentials were not provided." Not logged in
403 "You have not purchased this lecture." Student didn't buy
403 "Your access to this lecture has expired." Purchase expired
403 "You have reached the maximum watching limit for this lecture." Watch count exhausted
403 "You must pass the homework (ID X) before accessing this video." Prerequisite not met
403 "You can only preview videos in your own courses." Teacher trying to preview another teacher's video

POST /webhooks/bunny-encoded/

Description: Called by Bunny Stream when video encoding is complete. Updates the video's bunny_status to 4 (Finished).

Authentication: HMAC signature verification (backend validates request authenticity)

Request Body (Bunny sends this):

{

  "Status": "integer - New status"
}

Success Response — 200 OK:

{
  "status": "updated"
}

5. Watching Times Limit

Each lecture enforces a watching times limit (default: 4). Once a student has used all their watching times, the lecture's videos are locked.

Detail Value
Scope Per-student, per-lecture (watching any video counts)
Default 4
Max reopens 2 (each grants 1 extra day, does NOT reset watch count)
Locked access Homework, quizzes, materials still accessible
Teacher view watch_count and max_watch_count available in purchase details
