Removed Endpoints

Enrollment Management — 8 endpoints removed

Old Endpoint Method What it did Replacement
/courses/enrollments/pending/ GET List pending enrollments for teacher's courses GET /courses/enrollments/?status=pending
/courses/<id>/enrollments/pending/ GET Pending enrollments for one course GET /courses/enrollments/?course=X&status=pending
/courses/<id>/enrollments/approved/ GET Approved enrollments for one course GET /courses/enrollments/?course=X&status=approved
/courses/<id>/enrollments/rejected/ GET Rejected enrollments for one course GET /courses/enrollments/?course=X&status=rejected
/courses/<id>/students/ GET Approved students with balance GET /courses/enrollments/?course=X&status=approved (now includes balance field)
/courses/enrollments/my/ GET Student's own enrollments GET /courses/enrollments/ (already filters by student)
/courses/enrollments/bulk-action/ POST Bulk approve/reject enrollments POST /courses/enrollments/approve/ with enrollment_ids: [...]
/accounts/students/<id>/status/ PATCH Update student status (separate endpoint) PATCH /accounts/students/<id>/ with status, is_active, reason fields

Why: All enrollment GET endpoints could be replaced by the general GET /courses/enrollments/ which already supports ?status= and ?course= filters. The bulk-action was merged into the approve/reject endpoints by accepting an enrollment_ids array instead of requiring a dedicated endpoint. The status update was merged into the student detail update to consolidate all student modifications in one place.

Course Content — 3 endpoints removed

Old Endpoint Method What it did Replacement
/courses/<pk>/topics/ GET List topics for a course GET /courses/topics/?course=X
/courses/topics/<pk>/lectures/ GET List lectures for a topic GET /courses/lectures/?topic=X
/courses/lectures/<pk>/videos/ GET List videos for a lecture GET /courses/videos/?lecture=X

Why: These were convenience shortcuts that duplicated the general list endpoints. The general list views (TopicListCreateView, LectureListCreateView, VideoListCreateView) already support ?course=, ?topic=, and ?lecture= query parameters and return exactly the same data using the same serializers. One less URL to maintain, zero functional loss.

Student Reports — 2 endpoints removed

Old Endpoint Method What it did Replacement
/courses/lectures/<pk>/student-report/ GET Per-lecture student progress report Get data individually from enrollment list + homework submissions + quiz results + purchase list
/courses/<course_pk>/student-report/<student_id>/ GET Per-student full course report GET /payments/transactions/?student=X for balance history, enrollment list for purchases

Why: These were monolithic report endpoints that returned large amounts of data. The same information can be composed from simpler, reusable endpoints.

Balances & History — 3 endpoints removed

Old Endpoint Method What it did Replacement
/payments/balance/my/ GET Student's own balances GET /payments/balance/ (already filters by student)
/payments/balance/students/ GET Teacher views student balances GET /courses/enrollments/?course=X&status=approved (now includes balance field)
/payments/history/ GET Student activity log GET /payments/transactions/?student=X (full ledger with balance snapshots)

Why: The /my/ suffix added only a 400 error for non-students. The general list already filters by student role. The history/ endpoint was a lightweight view of the same data available in transactions/ which has more detail (balance_before, balance_after).

Materials — 1 endpoint removed

Old Endpoint Method What it did Replacement
/materials/my/ GET Student's purchased lecture materials GET /materials/ (already filters by student's purchased lectures)

Why: The general list endpoint GET /materials/ already filters by purchased lectures for students. Same serializer, same data.

Teacher Dashboard — 1 endpoint removed

Old Endpoint Method What it did Replacement
/courses/my_courses/ GET Teacher's courses with enrolled/pending counts GET /courses/ (already filters by teacher, same annotations)

Why: The CourseListCreateView.get_queryset() already filters by teacher=user.teacher_profile for teachers, using the same serializer with the same annotations.

Changed Endpoint Behavior

POST /courses/enrollments/approve/

Before: Only accepted a single enrollment via URL parameter:

POST /courses/enrollments/<id>/approve/
Body: { "response_note": "..." }

After: Accepts both single and multiple enrollment IDs:

POST /courses/enrollments/approve/
Body: { "enrollment_ids": [1, 2, 3], "response_note": "..." }

Response:

{
  "processed": 3,
  "total_requested": 3,
  "action": "approved",
  "errors": []
}

POST /courses/enrollments/reject/

Same change as approve — accepts enrollment_ids array instead of a single PK.

PATCH /accounts/students/<id>/

Before: Could only update profile fields (name, phone, grade, etc.). Status changes required a separate endpoint.

After: Now accepts status, is_active, and reason fields in addition to all profile fields. Includes full transition validation and audit log:

Allowed Transitions From To
pending verified, declined
verified suspended_temporary, suspended_permanent
declined verified
suspended_temporary verified
suspended_permanent verified

Reason is required for declined, suspended_temporary, and suspended_permanent. A StudentStatusLog entry is automatically created on every status change.

PATCH /accounts/teachers/<id>/

Before: Could not change password.

After: Now accepts an optional password field (min 8 chars). When provided, the teacher's password is updated via user.set_password(). The teacher can also use the forgot-password flow themselves.

Enrollments now validate school type and division

Student enrollment now checks that the student's school_type is in the teacher's subject's school_types and the student's division is in the teacher's subject's divisions (only enforced if the subject has school_types/divisions assigned — if none, any school_type/division is accepted).

Quiz questions can be created inline

POST /learning/quizzes/ and PUT/PATCH /learning/quizzes/<id>/ now accept an optional questions array with inline question text, type, order, points, and choices:

{
  "lecture": 5,
  "title": "Quiz Title",
  "settings": { ... },
  "questions": [
    {
      "text": "What is 2+2?",
      "question_type": "mcq_single",
      "order": 1,
      "points_override": 2,
      "choices": [
        {"text": "4", "is_correct": true, "order": 1},
        {"text": "5", "is_correct": false, "order": 2}
      ]
    }
  ]
}

Sending questions on update replaces all existing questions.

Assistant list now auto-filters by teacher

GET /accounts/teacher/assistants/ — teachers now see ONLY their own assistants. SiteOwner sees all. Assistants see coworkers under the same teacher.

QuizSettings — open_date/close_date removed

Quizzes no longer have open/close dates. Only Exams have open_date and close_date (directly on the Exam model). Timer max is now 1440 minutes (24 hours).

New Features Added

Feature Endpoint What it does
Balance Transaction Ledger GET /payments/transactions/ Full ledger with balance_before and balance_after for every code redemption and lecture purchase. Filterable by ?student=X, ?course=X, ?transaction_type=code_redeemed.
Thumbnail URL In all video responses thumbnail_url field automatically generates the Bunny Stream thumbnail URL from the video's GUID.
Bunny video deletion DELETE /courses/videos/<id>/ Now also deletes the video from Bunny Stream, not just the local database.
Auto-order Topics, Lectures, Videos When creating without specifying order, automatically assigns max_order + 1 among siblings.
Disk space check POST /courses/videos/<id>/upload/ Checks available temp disk space before accepting upload. Returns 503 if space is insufficient.
Password reset for teachers PATCH /accounts/teachers/<id>/ SiteOwner can reset a teacher's password directly.

Bug Fixes & Security

Fix Detail
Assistant isolation Teachers could see ALL assistants in the system. Now filtered to their own.
Bunny upload SSL retry Retries upload 3 times with exponential backoff on SSL errors.
Bunny auto-recreate on 404 If Bunny Stream auto-deleted the video entry, it's re-created automatically before upload.
Content-Length in upload Bunny Stream doesn't support chunked transfer encoding. Now sends explicit Content-Length header.
Stream from disk (0 RAM) Files are streamed from disk to Bunny via urllib3, never loaded into memory.
OTP expiry increased Changed from 10 to 30 minutes for student registration flow.
Enrollment phone validation Student phone number cannot be same as father's or mother's number.
Enrollment grade validation Student's grade must match course's grade. School type and division also checked if subject has them.
Status in me endpoint Students get status field to differentiate pending vs declined vs suspended.
Bunny upload to collection When Bunny entry is re-created (404), collection_id is passed so the video goes to the teacher's collection.
